# Tradvance — requirements.md

Claude Code가 이 문서를 읽고 구현을 시작할 수 있도록, 데이터 분석부터 해결안 도출, 검증 계획까지의 전 과정을 정리했다.

> **문서 구성 노트**: 이 문서는 직전 개인 과제("장바구니 전환율 개선")에서 받은 피드백 세 가지를 의도적으로 반영해 구성했다. ① 문제 정의에는 해결책을 섞지 않고 현상/원인가설/근거 3단으로 분리한다. ② 우선순위 판단은 정성적 표현("우려가 크다") 대신 실제 응답 빈도 숫자로 뒷받침한다. ③ 데이터에서 발견했지만 이번 범위에 쓰지 않는 내용은 "왜 안 썼는지"를 명시한다.

---

## 1. 데이터 분석

### 분석 목적
해외 바이어 문의 응대에서 소규모 제조업체 담당자가 겪는 어려움과, AI 도구 도입 시 우려하는 지점을 확인한다.

### 1) 답변 지연 원인 (인터뷰 9명, 자유응답 집계)
- 내부 담당자 확인 반복
- 견적 조건 확정의 어려움
- 영어 표현 부담

### 2) 확인 항목 언급 빈도
가격/견적이 가장 많이 언급되었고, 인증/품질 기준, 납기/리드타임, MOQ, 샘플, 배송/물류, 결제조건 순으로 뒤를 이었다. 이 7개 항목을 그대로 확인 항목 카테고리로 채택했다.

### 3) 선호 응대 방식 (9명 응답 재집계)

| 선호 방식 | 응답자 수 |
|---|---|
| AI 초안 + 담당자 검토 후 발송 | 5 |
| AI는 확인 항목만 정리, 답변은 사람이 작성 | 3 |
| AI가 답변까지 자동 발송 | 1 |

→ 9명 중 8명이 완전 자동화가 아닌 "사람 개입 구조"를 선택했다. 이 결과가 승인 구조(발송 없이 복사만) 설계의 직접적 근거다.

### 4) AI 도입 시 우려 항목 (재검토 — 응답 원문 재집계)

기존에 "잘못된 견적/납기 안내가 최다 우려"로 정리했으나, 응답 원문을 다시 세어보니 정확한 1위는 아래와 같다. 이 부분을 숫자로 재확인해 바로잡는다.

| 우려 항목 | 언급 응답자 수(9명 중) |
|---|---|
| AI 답변을 그대로 믿고 발송할 위험 (과신) | 6 |
| 잘못된 견적 안내 | 4 |
| 기술 사양 오역 | 3 |
| 회사 내부 정보 유출 | 3 |
| 납기·재고 오안내 | 2 |
| 특별히 우려 없음 | 1 |

→ 1위는 "AI 답변 과신"이다. 이는 견적·납기·사양 오류를 포함한 **상위 개념의 우려**이므로, 위험 구간 체크리스트 기능이 이 1위 우려를 정확히 겨냥한다는 논리는 그대로 유지된다. 다만 "견적/납기가 최다 우려"라는 이전 표현은 부정확했으므로 이 문서부터 정정한다.

### 5) 보조 통계 (외부 자료)
전경련중소기업협력센터 조사(2018, 1,020개사 대상)에서 응답 기업의 34.6%가 바이어·파트너 발굴의 어려움을 애로사항으로 꼽았다. 이는 **발굴 단계** 애로이며 본 프로젝트가 다루는 **응대 단계**와는 인접 문제일 뿐 직접 일치하지 않는다. 참고용 배경 정보로만 사용하고, 핵심 근거로는 인터뷰 데이터를 우선한다.

### 6) 흥미로운 발견이지만 이번 범위에서 사용하지 않는 것
수출 실무 경력자(응답자 1명, 은퇴)는 "계약 클로징에 가까운 신뢰 구간은 사람 대 사람 소통이 필수이며, 이 구간의 LLM 완전 자동화는 리스크가 크다"는 풍부한 의견을 제공했다. 이는 문의 응대(1차 정보성 소통) 단계를 다루는 이번 MVP 범위 밖의 통찰이라 핵심 문제 정의에는 반영하지 않았다. 대신 10장(리스크)에서 "의도적 범위 제외" 사유로 명시한다. (이전 과제 피드백의 "가격대 분석처럼 흥미로운 발견을 이후 논리와 연결 못한 문제"를 반복하지 않기 위한 조치.)

## 2. 문제 정의

### 문제 현상
수출 담당자가 부족한 소규모 제조업체는 해외 바이어 문의를 받았을 때 확인해야 할 항목이 많아 답변 작성이 지연되고, 확정되지 않은 정보를 실수로 확정처럼 전달할 위험이 있다.

### 원인 가설
담당자가 문의를 받은 즉시 스스로 확인 항목을 판단하고, 내부 확인을 반복하며, 확정 여부를 표시해주는 장치 없이 답변을 작성하기 때문이다.

### 데이터 근거

| 근거 | 수치 |
|---|---|
| 완전 자동화가 아닌 "사람 개입 구조" 선호 | 9명 중 8명 |
| AI 답변 과신에 대한 우려 | 9명 중 6명 |
| 확인 항목 최다 언급 | 가격/견적 (7개 카테고리 중 1위) |

이 표는 해결책이 아니라 문제가 실재하고 그 방향이 명확하다는 근거만 담는다. 해결안은 5장에서 별도로 다룬다.

## 3. 가설 수립

**검증 가능한 가설**: 바이어 문의를 입력했을 때 확인 필요 항목과 위험 구간이 표시된 답변 초안을 제공하면, 담당자는 더 빠르고 안전하게 응대할 수 있을 것이다.
→ 확인 항목 정리 체감도가 높아질 것이다.
→ 위험 구간 체크리스트에 대한 실무 유용성 체감도가 높아질 것이다.

## 4. 경쟁사 벤치마킹

### 직접 경쟁자 — 린다(Rinda, 그린다에이아이)

자율형 AI 영업 에이전트로, 웹사이트 주소만 입력하면 바이어 발굴부터 다국어 이메일 발송까지 자동 수행한다. 제조·화장품·식품을 명시적으로 타겟팅하고 있어 ICP가 겹친다.

| 구분 | 린다(Rinda) | Tradvance |
|---|---|---|
| 핵심 축 | 발굴 이후 아웃바운드 자동화(콜드메일) | 바이어 문의 **이후**의 응대(견적·납기·사양) |
| 자동화 수준 | 완전 자율형(사용자 개입 최소화) | AI는 초안까지, 최종 행동은 담당자가 직접 |
| 산업 커버리지 | 제조·화장품·식품 등 범용 | 산업용 부품·기계·원자재 특화 |
| 위험 관리 | 회사 스스로 "계약 기여도, 산업별 관행 반영의 정교함은 검증 단계"라 인정(약관 명시) | 위험 구간을 명시적으로 표시, 체크 전까지 복사 불가 |

### 국내 인접 사례 (요약)

| 기업 | 성격 | Tradvance와의 관계 |
|---|---|---|
| EC21 | 국내 최대 B2B 무역 플랫폼, 발굴+컨설팅 | 발굴 단계, 응대 단계는 다루지 않음 |
| 고비즈코리아 | KOTRA 운영 온라인 수출 지원 플랫폼 | 정부 채널 위에서 못하는 응대 자동화에 집중 |
| 한국산업단지공단 제조거래센터 | 국내 제조업체 간 B2B 매칭 | 장기 로드맵과 겹치는 지점, 이번 MVP와 무관 |

### 해외 사례 (3개 그룹 요약)
- **발굴형 마켓플레이스**(Alibaba, Kompass 등): 발굴 단계, Tradvance와 겹치지 않음.
- **AI 세일즈 자동화**(Apollo.io, Outreach.io, Instantly.ai 등): 콜드 아웃바운드·리드 규모 경쟁 중심. **산업재 수출 응대에 특화된 사례 없음** — 포지셔닝 근거.
- **무역 실행·물류**(Flexport, e2open 등): 발굴이 아닌 실행 이후 복잡성 자동화, 이미 대기업 시장에서 M&A로 수렴.

**벤치마킹 시사점**: 발굴과 실행관리는 국내외 모두 포화 상태에 가깝다. 응대 자동화 영역에서도 "산업재 수출 문의 응대"라는 좁은 문제에 특화된 사례는 확인되지 않았다. 이 빈자리가 Tradvance의 해결방안(5장)의 존재 이유다.

## 5. 해결방안

### 기능명
위험 구간 체크리스트가 포함된 응대 초안 생성

### 기능 목적
바이어 문의 원문을 입력하면 확인 항목을 분류하고, 확정되지 않은 정보를 담당자가 실수로 확정처럼 전달하지 않도록 위험 구간을 짚어주는 영어 답변 초안을 제공한다.

### 기능 설명
- 확인 항목 7개 카테고리로 분류 (가격/견적, MOQ, 납기, 기술 사양, 포장/물류, 인증/품질 기준, 결제 조건)
- 정규식(숫자·통화·기간 패턴) + AI 설명 결합한 하이브리드 방식으로 위험 구간 탐지
- 위험 구간은 항목별 인용문 + 위험 사유로 체크리스트 표시
- 전부 체크해야 "복사하기" 활성화, 발송 기능 없음

### 기능명세서

| 항목 | 내용 |
|---|---|
| 입력폼 | 텍스트박스 (바이어 문의 원문 붙여넣기) |
| 버튼 | "분석 시작하기", "위험 구간 확인하기", "초안 복사하기" |
| 버튼 활성화 조건 | 입력창에 텍스트 존재 시 분석 가능. 위험 구간 전부 체크 시 복사 가능 |
| 버튼 비활성화 조건 | 입력 없음, 또는 위험 구간 미확인 상태 |
| 로직 | 원문 → 7개 카테고리 태그 분류 → 영어 초안 생성 → 위험 구간 정규식+AI 탐지 |
| 예외 상황 | 바이어 문의로 인식 안 됨 → 재입력 안내. 위험 구간 0건 → 체크리스트 생략, 바로 복사 가능 |
| 제한 개수 | 없음 (탐지된 위험 구간 전부 표시) |
| 외부 액션 | 없음 (발송 기능 없음, 클립보드 복사까지만) |
| 측정 이벤트 | 분석 시작, 위험 구간 확인 모달 오픈, 체크 완료, 복사 완료 |

## 6. 왜 이 해결방안을 선택했는가

1번(데이터 분석)에서 확인한 수치를 그대로 근거로 사용한다.

| 근거 | 수치 | 해석 |
|---|---|---|
| 사람 개입 구조 선호 | 9명 중 8명 | 완전 자동 발송이 아닌 승인/복사 구조가 맞다 |
| AI 답변 과신 우려 | 9명 중 6명(1위) | 위험 구간을 짚어주는 기능이 최다 우려를 정면으로 겨냥 |
| 가격/견적 확인 항목 최다 언급 | 7개 중 1위 | 확인 항목 분류 체계의 타당성 |

경쟁사 분석(4장)에서 확인한 "산업재 응대 자동화의 공백"과 결합하면, 이 해결방안은 데이터 근거와 경쟁 환경 근거를 모두 설명할 수 있는 방향이다.

**왜 이메일 자동 연동이 아니라 붙여넣기인가**: 담당자가 이미 쓰는 메일함에서 벗어나지 않게 하기 위함이며, 인증 절차 없이 즉시 사용 가능해야 한다.

**왜 7개 카테고리 전체에 하이브리드 탐지를 적용하는가**: 개인 프로젝트로 진행하며 발생 가능한 안정성 리스크를 인지한 상태에서, 범위를 축소하지 않고 전체 카테고리를 유지하기로 결정했다. 데모 전 반드시 샘플 3개로 재현 테스트를 거친다.

## 7. AS-IS → TO-BE

| 구분 | AS-IS | TO-BE |
|---|---|---|
| 문의 접수 후 첫 행동 | 담당자가 스스로 확인 항목 판단 | 붙여넣기 → AI가 7개 항목으로 즉시 분류 |
| 내부 확인 | 여러 번 내부 담당자에게 반복 문의 | 확인 필요 항목이 먼저 정리되어 헤매는 시간 감소 |
| 답변 작성 | 영어 표현 부담 + 백지에서 작성 | AI가 초안 생성, 담당자는 확인·수정만 |
| 실수 방지 | 담당자 개인 주의력에 의존 | 위험 구간 자동 표시, 체크 전까지 복사 불가 |
| 최종 발송 | 담당자가 직접 작성해 발송 | (동일) 담당자가 직접 발송, AI는 발송하지 않음 |

## 8. 화면 구조 초안

단일 흐름 5화면(상세는 별도 와이어프레임 PDF 참고):
1. 문의 입력 (로그인/대시보드 없음)
2. 분석 결과 · 답변 초안 (세로 아코디언, 완료 단계도 펼친 채)
3. 위험 구간 체크리스트 모달 (상태 A 확인 전 / 상태 B 확인 완료)
4. 로딩 상태
5. 예외 상태 (문의 미인식, 위험 구간 0건, 복사 완료 토스트)

## 9. KPI

### KPI 산정 근거
이 프로젝트는 실제 서비스 트래픽이 없는 미니 프로젝트이므로, 장바구니 전환율 같은 실측 지표 대신 **박람회 부스 즉석 검증**을 기준으로 삼는다. 이 점에서 실제 사용자 데이터를 가진 서비스의 KPI 산정과는 근본적으로 다르다는 것을 명시한다.

| 구분 | 지표 | 측정 방법 | 목표 |
|---|---|---|---|
| Primary | 확인 항목 유용성 체감 | 5점 척도 즉석 질문 | 평균 3.5 이상 |
| Secondary | 위험 구간 체크리스트 유용성 체감 | 5점 척도 즉석 질문 | 평균 3.5 이상 |
| Guardrail | 흐름 완주율 | 입력→복사까지 도달 여부 | 부스 시연 중 이탈 없음 |
| Guardrail | 위험 구간 탐지 정확도(QA) | 샘플 3개 × 5회 재현 테스트 | 5/5 안정 탐지 |

## 10. 검증 계획

### A. 기술 신뢰성 테스트 (QA)
샘플 3개(MOQ/납기/인증 중심) 각각 5회 반복 실행, 위험 구간 탐지 결과 기록. 배포 전날 + 데모 직전 1회 추가 수행.

### B. 사용자 인지 검증 (박람회 부스)
- 5점 척도 2문항 + 개방형 1문항("가장 걱정되는 점", 인터뷰 문항 재사용)
- 표본: 부스 방문자 최소 10명
- 한계: 응답자가 실제 제조업 수출 담당자가 아닌 과정 수강생

### 실험 결과 판단 기준표

| 위험구간 탐지 정확도(QA) | 사용자 체감 점수 | 판단 | 액션 |
|---|---|---|---|
| 안정적 (5/5) | 3.5 이상 | 가설 검증 | 그대로 데모 진행, 확장 로드맵 함께 소개 |
| 안정적 (5/5) | 3.5 미만 | 기능은 되나 체감 약함 | "체감보다 QA로 검증된 안정성"을 강조하는 방향으로 발표 전략 조정 |
| 불안정 (5회 중 3회 이하) | 3.5 이상 | 컨셉은 통하나 구현 리스크 | 데모는 사전 검증된 샘플만 사용, 회고에 기술적 한계 명시 |
| 불안정 | 3.5 미만 | 재설계 필요 | 정규식 단독 방식으로 축소, AI 설명 없이 하이라이트만 유지 |

### 향후(로드맵) A/B Test 설계 — 이번 미니 프로젝트 범위 밖
실 서비스로 확장될 경우를 가정한 설계만 남겨둔다. 이번 제출물에는 적용하지 않는다.

| 항목 | 내용 |
|---|---|
| 대상 | 실제 문의 응대 담당자 |
| A안 | 위험 구간 표시 없는 일반 AI 초안 |
| B안 | 위험 구간 체크리스트 포함 초안 |
| Primary KPI | 초안 수정 없이 그대로 사용하는 비율(오히려 낮아야 안전, 즉 체크 후 수정률로 대체 측정) |
| Guardrail KPI | 위험 구간 오탐지율, 사용자 이탈률 |

## 11. 예상 결과

이 프로젝트는 실사용 트래픽 데이터가 없으므로, 장바구니 전환율 개선 과제처럼 "몇 명 → 몇 명"의 정량 전환 예측치를 제시하지 않는다. 근거 없는 숫자를 만들어내는 것은 이전 과제 피드백에서 지적된 "감으로 나온 목표 수치" 문제를 그대로 반복하는 것이기 때문이다. 대신 인터뷰에서 확인된 수용도를 기대치로 제시한다.

| 항목 | 근거 | 기대 |
|---|---|---|
| 사람 개입 구조 수용도 | 9명 중 8명이 이미 선호 | 부스 검증에서도 유사한 긍정 반응 기대 |
| 위험 구간 체크리스트 체감 | 9명 중 6명이 AI 과신을 최다 우려로 꼽음 | 이 우려를 정면으로 해소하므로 체감 점수 3.5 이상 기대 |

## 13. 프로젝트 한계

- 인터뷰 표본이 9명으로 크지 않고, 제조업·수출 실무 경험자 비중이 응답자 전체는 아니다.
- 박람회 부스 검증 대상은 실제 제조업 수출 담당자가 아니라 같은 과정 수강생이다.
- 실사용 트래픽 데이터가 없어 정량적 전환율 예측(11장)을 제시하지 않았다 — 이는 한계이자, 근거 없는 수치를 만들지 않기 위한 의도적 선택이다.
- 위험 구간 탐지는 정규식+AI 하이브리드로 설계했으나, 실제 다양한 문의 유형에 대한 정확도는 QA 재현 테스트 범위(샘플 3개)를 넘어서는 검증은 하지 못했다.
- 계약 클로징 등 신뢰가 결정적인 응대 단계는 다루지 않았다 (10:58 인터뷰 응답자 의견 기반, 의도적 제외).

## 14. 최종 결론

수출 담당자가 부족한 소규모 제조업체는 해외 바이어 문의 응대에서 확인 항목 과다와 오안내 위험을 동시에 겪고 있으며, 인터뷰 데이터상 9명 중 8명이 완전 자동화가 아닌 사람 개입 구조를, 6명이 AI 답변 과신을 최다 우려로 꼽았다. 경쟁사(린다 등)는 발굴·아웃바운드 자동화에 집중하고 있어, 문의 이후 응대와 위험 관리라는 영역은 국내외 모두 비어 있는 자리다. 이를 바탕으로 "위험 구간 체크리스트가 포함된 응대 초안 생성" 기능을 MVP로 제안하며, QA 재현 테스트와 박람회 부스 즉석 검증을 통해 결과를 확인할 계획이다.

## 15. 제작 회고 초안 (브로셔용)

**잘된 점**: 처음엔 수출 전 과정을 지원하는 넓은 서비스로 기획했으나, 설문·인터뷰·경쟁사 분석을 거치며 "문의 응대"라는 좁고 검증 가능한 문제로 좁혔다. 우려 항목도 감이 아니라 응답 재집계를 통해 순위를 바로잡았다.

**아쉬운 점**: 인터뷰 규모가 크지 않고 부스 검증 대상이 실제 제조업 수출 담당자가 아니라는 한계가 있다. 실사용 데이터가 없어 정량적 전환율 예측은 제시하지 못했다. 계약 클로징처럼 사람 대 사람 신뢰가 결정적인 구간은 의도적으로 이번 범위에서 제외했다.

**다음에는 이렇게 할 것**: 실제 제조업 담당자를 대상으로 한 검증을 늘리고, 위험 구간 탐지 로직을 정량적으로 평가하는 절차를 더 이른 단계에 마련하겠다.
